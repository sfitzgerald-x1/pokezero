# CPU Self-Play Roadmap

This document summarizes the current state of the local CPU-based self-play proof of concept and the remaining work needed before it should be treated as an experiment loop rather than scaffolding.

## Goal

The near-term goal is a local, CPU-only Gen 3 random battle loop that can:

- generate teacher-bootstrap and self-play data through the local Showdown harness
- train and resume linear baseline policies without GPU dependencies
- benchmark candidates against fixed baselines and promoted historical checkpoints
- gate checkpoint advancement with auditable promotion criteria
- stop bad long-running jobs through post-iteration audits
- compare completed runs with enough metadata to decide what to try next

This goal is narrower than the eventual transformer/PPO direction. The CPU loop is successful when it can produce reproducible, benchmarked runs that are safe to leave running locally and informative enough to compare.

## Current State

The CPU loop is now mostly wired end to end.

Implemented:

- The local Showdown BattleStream environment can run Gen 3 random battles and expose stable player-relative observations for either side.
- The rollout path records JSONL trajectories with legal masks, selected actions, opponent-action labels, terminal outcomes, capped-game markers, policy identifiers, and collection metrics.
- Dataset loading supports temporal windows and terminal-winner-derived returns, so winning actions are reinforced even when the immediate step reward is zero.
- A dependency-free linear softmax policy can train from behavior-cloning or reward-weighted objectives and save/load versioned checkpoints.
- A scripted Gen 3 randbat teacher exists for bootstrap data. It uses local Showdown dex metadata and first-pass context heuristics for damage preference, status pressure, team-status cures, low-HP preservation switches, status-aware switch penalties, recovery/setup, Spikes, and Rapid Spin. These heuristics shape bootstrap examples only; they are not a permanent hand-coded strategy layer for the learned policy.
- Teacher bootstrap can collect train and held-out validation rollouts, optionally add deterministic teacher-scenario demo records for sparse branches to training only, run strict-teacher preflight, train a linear checkpoint, benchmark it, and write a manifest.
- Linear self-play can run multiple iterations from random, simple, or checkpoint initial policies with current-policy-only training data, frozen historical opponents, checkpoint warm starts, resume support, validation metrics, and parallel collection workers.
- Linear checkpoints omit the default all-zero opponent-action head, and legacy checkpoints with serialized all-zero opponent heads are compacted while loading. This reduces the memory cost of policy/opponent factory construction without changing action selection.
- Linear self-play collection reuses the already-loaded current linear model when building the current-policy factory, instead of deserializing the current checkpoint a second time before collection. Opponent checkpoint factories still load from their checkpoint specs.
- Neural scaffolding exists behind the optional `neural` extra, including offline transformer training, neural checkpoint benchmarking, and a first neural iteration loop. This is not required for the CPU baseline milestone.
- Promotion gates check benchmark win rates, minimum game counts, incumbent comparisons, capped-game rates, teacher-degradation counters, and Wilson lower-bound no-regression criteria.
- The promotion registry records accepted checkpoints, can copy managed artifacts, verifies registry/checkpoint integrity, feeds promoted opponent pools back into self-play, previews pool selection, advances auto-promoted managed artifacts as the next current policy spec, excludes the current checkpoint by resolved checkpoint identity rather than raw string equality, and supports recoverable retention archiving.
- Run audits can check latest benchmark health, capped-game rates, same-opponent regressions, repeated promotion failures, decision-round length, missing benchmark opponents, and best-effort process RSS high-water marks.
- Linear self-play iteration manifests record process peak RSS high-water snapshots at key phases, and run audits include those phase snapshots in `latest_process_peak_rss_mb`. These snapshots use the process-lifetime RSS high-water counter, so they show where a new lifetime peak is first reached rather than per-phase current memory.
- Self-play collection metrics now record nested process peak RSS snapshots for the collection window, including policy-factory creation, output setup, record collection progress, output commit, and summary. Run audits surface those collection-phase snapshots for diagnosis while the top-level process high-water snapshots remain the RSS ceiling signal.
- Run audits can mark selected checks as warning-only diagnostics, so early CPU runs can surface provisional decision-round, same-opponent-regression, or experiment-scoped RSS concerns without stopping when the strict benchmark/capped/coverage gates still pass.
- Rollout benchmarking can compare arbitrary candidate policies against a shared opponent set in mirrored seats, with optional candidate-vs-candidate mirrors. This provides the command shape needed for direct cold-start versus teacher-bootstrap evidence instead of aggregate comparisons with different opponent sets.
- CPU smoke, pilot, and long-run wrappers can generate plans, execute guarded runs, persist wrapper summaries, calibrate audit configs from pilot evidence, replay those configs, launch readiness-checked long runs, and report or compare run health from summaries.
- Self-play and CPU long-run wrappers support an explicit runtime-health audit failure mode for capacity diagnostics. In this mode, runtime-health failures still stop the run, while promotion-strength failures remain visible in reports without stopping the wrapper.
- CPU smoke and pilot execution wrappers now reject non-fresh run roots before launching nested work, so accidental reruns do not overwrite wrapper summaries or collide later with existing bootstrap/self-play artifacts.
- CPU pilot planning and execution now reject calibration benchmark-game floors that the selected `--evaluation-games` cannot guarantee before launching nested work.
- CPU long-run planning can decouple the promotion gate profile from the post-iteration audit source, so a feasible smoke/default gate profile can still enforce the stronger pilot-derived or summary-derived audit config during the run.
- CPU long-run reports and comparisons split derived audit status into runtime-health and promotion-strength fields, so a capacity-valid run that fails to advance a checkpoint can be read without conflating those outcomes.
- `cpu-readiness-report` can roll up core pilot readiness, long-run derived audit health, and promotion-registry opponent-pool readiness from existing artifacts without launching games.
- Source provenance is recorded in major run artifacts so dirty or unexpected code snapshots are visible during review.

## What Is Left

The remaining work is less about wiring and more about making the loop empirically usable.

1. Continue calibrating hard-stop versus diagnostic guardrails from real expanded runs.

   The first summary-based envelope config was useful because it stopped runs that violated explicit health gates, but it was calibrated from only two smoke-profile summaries. Warning-only audit checks now let decision-round ceilings and same-opponent regression remain visible diagnostics while benchmark win-rate, capped-game, benchmark coverage, promotion-failure, and RSS checks stay eligible as hard stops. More evidence is still needed before these warning boundaries should be promoted back into hard failures. Validation fit remains imitation-health only.

- Run at least two self-play iterations with `games-per-iteration >= 32`, `workers = 4`, `feature-count = 131072`, `window-size = 4`, and `evaluation-games >= 30`.
- Require a promoted historical opponent pool of at least one selectable checkpoint before collection starts.

- Do not replace `expanded-teacher-bootstrap-linear-iter-0001` as the working reference from wrapper aggregate metrics alone.
- Require a targeted shared-opponent benchmark before claiming a candidate is stronger than the current reference.

## Suggested Next Tasks

The next implementation tasks should be chosen in this order unless a real pilot run exposes a more urgent failure.

1. Keep the existing standalone teacher-bootstrap reference. The first measured scenario-demo recipe did not clear it. If continuing on data quality, change the recipe deliberately, for example demo weight/selection, rollout volume, or training regularization, before rerunning shared-seed comparison; same-teacher/same-recipe reruns should be treated as variance checks, not new reference candidates.
2. Treat the passing four-iteration threshold run as the current CPU run-health baseline for this exact seeded promoted-opponent-pool shape. If capacity work continues, extend the same runtime-health shape again or vary one dimension deliberately before raising or relaxing the RSS ceiling.
3. If choosing data-quality work, target sparse teacher branches or an intentionally different bootstrap recipe before another reference comparison.
4. Use targeted shared-opponent benchmarks, not wrapper aggregate metrics, before replacing the current working reference.
