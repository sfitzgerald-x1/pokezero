Embedding Space & Efficiency Targets

Where PokeZero's neural parameters actually live — and how to spend them better.

Analysis · 2026-06-23 · based on the goal-run checkpoint neural-teacher-bc-probe and src/pokezero/showdown.py
TL;DR. The model is reported as 128.3M parameters / 513 MB, but 99.8% of that is one embedding table — a generic 1,000,001-bucket hash shared across every categorical feature. The actual transformer is only ~270K params. The hash table is enormous, semantically structureless, almost entirely untrained on 1,922 examples, and it throws away Gen 3 random battle's biggest advantage: a small, known, finite vocabulary. Replacing it with explicit per-field embeddings is a ~100× size reduction that should also improve learning.

2. How the embedding space works today

Every categorical value in an observation — a species, move, item, ability, status, weather, belief fact, etc. — is turned into an integer id by a single shared hash encoder:

def stable_category_id(value, *, buckets=1_000_000):
    normalized = str(value or "").strip().lower()
    if not normalized: return 0                      # 0 = padding
    digest = hashlib.blake2b(normalized.encode(), digest_size=8).digest()
    return (int.from_bytes(digest, "big") % buckets) + 1   # 1 .. 1,000,000

That id indexes one nn.Embedding(1_000_001, 128) table. The observation is 46 tokens (1 field + 6 self + 6 opponent + 9 action candidates + 24 recent events) × 84 categorical columns each (4 base fields + 80 belief-fact buckets) × a 4-turn history window — and every one of those columns looks up the same shared table.

The source even labels this as temporary: “a stable hash-bucket encoder for early experiments. Explicit vocabularies can replace it once the observation vocabulary is finalized.” (showdown.py:341) — this document is that follow-up.

3. Why it is inefficient

  • Wildly over-provisioned for the data. 1,000,001 rows trained on 1,922 examples: the vast majority of rows were never touched and sit at their initialization. The 513 MB checkpoint is mostly dead weight.
  • No semantic structure. Hashing scatters related concepts to unrelated random rows — "Charizard" and "Blastoise", or "Surf" and "Hydro Pump", share nothing. Every concept must be learned from scratch with no inductive sharing.
  • Collisions. A 64-bit blake2b reduced mod 1,000,000 can map distinct values to the same row; the model cannot tell those apart.
  • It discards the format's core advantage. Gen 3 random battle entities come from a known, finite generator. We are paying for an open-vocabulary hash in a closed-vocabulary game.
  • Belief columns dominate width. 80 of the 84 categorical columns per token are belief-fact buckets, multiplying lookups into the same oversized table.

The true vocabulary is tiny and enumerable (Gen 3 totals; random-battle pools use subsets):

4. Efficiency targets (ranked)

1Explicit finite-vocabulary embeddings (biggest win)

Replace the single shared hash table with small, typed embeddings keyed on real ids — one table per field family (species, move, item, ability, status, type, …), each sized to its true cardinality plus padding/OOV. Total embedding rows drop from 1,000,001 to roughly a few thousand: a ~100–1000× reduction in embedding parameters. Every row now sees training signal (denser, better-trained representations), collisions vanish, and related fields can share a table (e.g. all four move slots use one move embedding) for inductive generalization. This directly operationalizes the "known finite set space" thesis from the architecture doc — and should improve sample efficiency, not just shrink the file.

2Shrink the hash space (cheap interim)

If keeping the hash encoder short-term, drop buckets from 1,000,000 to ~32K–65K. That still over-covers the true vocabulary by 30×+, but cuts the table ~15–30× immediately with a one-line change and a retrain. A stopgap, not the destination.

3Rethink the 80 belief-fact bucket columns

Belief facts are 80 of 84 categorical columns per token. Consider a compact learned belief embedding (or pooling possible-set ids through the shared field embeddings) instead of 80 wide bucket columns, reducing both lookups and observation width while preserving the belief signal.

4Checkpoint & precision hygiene (secondary)

Today's 513 MB file persists ~1M mostly-empty fp32 rows. With a real vocabulary the checkpoint falls to single-digit MB on its own. Lower-precision storage or persisting only touched rows are band-aids that target 1's by-product rather than the cause; prefer 1.

Separate axis — do not conflate. Growing the transformer trunk (more layers, larger dim, longer history window) is a different decision driven by data volume and opponent strength, and it carries CPU-inference cost. The embedding cleanup here is about spending parameters well, not about adding capacity.

6. Trade-offs & caveats

  • Out-of-vocabulary handling. Explicit vocabularies need a defined OOV/unknown id and a maintenance path when the dex or randbat pool changes. The hash "just works" for any string — that convenience is exactly what we trade for structure. In a fixed Gen 3 format this is a good trade.
  • Checkpoint compatibility. Changing the embedding scheme breaks existing neural checkpoints (a new schema/fingerprint). Acceptable — there is only one probe checkpoint today.
  • Efficiency ≠ guaranteed strength. The primary, near-certain win is size and trainability; the strength upside (denser embeddings, sharing) is likely but must be confirmed by the same shared-opponent benchmark methodology, against the current 0.77-vs-simple-legal result.
  • Validate, don't assume. Treat any change as a measured experiment: retrain on the same data, benchmark vs simple-legal/random-legal across two seed bands, compare.
