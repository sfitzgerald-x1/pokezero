PokeZero — Current Model Input Shape

What the neural policy actually sees per decision, after the lean-encoding cleanup (PR #215), the raw type/move-mechanic enrichment (Phase 1, PR #216), and the dynamic-state overhaul (Phase 2: level, base stats, boosts, weather, hazards/screens, PP, secondary-effect type+chance, turn count, volatile statuses; plus the party-slot token drop and right-sized belief buckets).

Input tensor per turn: 46 × (39 categorical ids + 37 numeric) + a token-type id and attention mask per token. Categorical ids are compact embedding-row indices (886 closed-vocab tokens + 16 OOV + padding = 903 rows). The 39 categorical columns: 9 fixed (primary / secondary / role / slot / type×2 / move-category / move-effect / move-priority) + 24 belief buckets (2 ability, 6 item, 16 move) + 6 active-mon volatile columns.

How a categorical id works (string → row, no hash). The strings below (species:Charizard, move:Flamethrower) are human-readable keys; each maps by direct lookup in the enumerated closed-universe vocabulary to a small embedding-row index. The model embeds the row directly — the legacy 1,000,000-bucket blake2b hash + in-model remap are retired:
"move:flamethrower" → CategoryVocabulary lookup → row 364 → 128-d embedding
What is the "field" token? Token 0 is a single global / context token (like a [CLS]) for battle-wide info not attached to a specific mon or move. It carries request_kind (move / force-switch / team-preview / wait) plus the global dynamic state — weather (categorical SECONDARY slot), player-relative hazard layers + screens for both sides, and the turn count (all numeric).

So the window reads, in order: "I clicked Flamethrower at their Skarmory → they switched to Blissey → Skarmory took damage → my Charizard got burned." This is the temporal signal that lets the model notice patterns like "the opponent keeps pivoting out of my Charizard" — the basis for anticipatory switching (hypothesis H2). Note the numeric magnitude of damage and exact HP live in the numeric features of the Pokémon tokens, not here.

Encode-time resolution. A few moves are context-dependent and resolved from the acting mon's state when the observation is built (a mon's type/HP are stable enough within a turn): base power for HP-variable moves (Reversal / Flail scale inversely with the user's HP; Eruption / Water Spout directly), and Curse's effect by the user's type (Ghost → move_effect:curse + 0.5 HP cost; non-Ghost → move_effect:curse_setup).
The model now sees who (species/move ids + types), how hurt (HP/status), what's still uncertain (belief), how hard / how fast (base power, base stats, level), the dynamic board (boosts, weather, hazards/screens, turn count, volatiles), and per-move tactics (PP scarcity, secondary-effect type + chance).

Note the shared type:fire row on both Charizard and Flamethrower — a single Fire embedding the model uses to learn STAB and effectiveness across every Fire mon/move at once. Other representation cleanups in this overhaul

ChangeRationale
Dropped the party-slot token (self_slot/opponent_slot on team tokens)team order is arbitrary in randbats → the index is unactionable, and the sequence position + token-type already identify the slot. (Action switch_slot/move_slot stay — they map to the emitted action.)
Right-sized belief buckets (ability 8→2, item 8→6, move 64→16) + positional, not hashedmeasured Gen 3 per-species maxima are 2 abilities / 5 items / 14 moves, so positional placement is exact and collision-free; the old hash collided badly and the old sizes were padding.
